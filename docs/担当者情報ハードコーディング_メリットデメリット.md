# 担当者情報をアプリにハードコーディングする場合のメリット・デメリット

担当者台帳（API: `GET /api/employees?code=...`）に毎回アクセスせず、アプリ内に担当者情報（コード⇔氏名）を直接持つ方式の比較です。

---

## メリット

| 項目 | 説明 |
|------|------|
| **オフライン・不安定環境でも利用可能** | ネット／API が使えない現場でも、担当者コード入力・氏名表示ができる。 |
| **応答が速い** | API 往復がなくなるため、担当者コード入力後の反映や伝票一覧の担当者表示が即時になる。 |
| **API／DB 負荷の軽減** | 担当者取得のリクエストがなくなるため、サーバー・担当者台帳テーブルへのアクセスが減る。 |
| **実装が単純** | `fetchEmployee` 呼び出しやエラーハンドリング（タイムアウト・通信失敗）を削減できる。 |
| **テスト・開発がしやすい** | API を立てなくても担当者コード入力〜伝票表示までの動作確認ができる。 |
| **運用コストの削減** | 担当者台帳の参照がアプリに閉じるため、API の可用性に依存しなくなる。 |

---

## デメリット

| 項目 | 説明 |
|------|------|
| **情報の更新にアプリの再配布が必要** | 担当者の追加・氏名変更・削除をするたびにアプリを更新・再インストールする必要がある。 |
| **台帳との不整合リスク** | 本社の担当者台帳とアプリの一覧がずれると、表示される氏名が古くなったり、存在しないコードが「有効」になったりする。 |
| **複数拠点・多数担当者で運用しづらい** | 担当者が多い／変動が大きい場合、ハードコーディングの一覧の更新・管理が煩雑になる。 |
| **セキュリティ・情報漏れ** | アプリに氏名一覧が含まれるため、APK 解析などで担当者リストが取り出される可能性がある（必要なら考慮が必要）。 |
| **バージョンごとに内容が違う** | 古いアプリと新しいアプリで「有効な担当者」が異なり、サポート時に混乱しやすい。 |
| **初期データの用意が必要** | コードと氏名の対応表をどこか（コード・設定ファイル・assets など）で用意・保守する必要がある。 |

---

## 運用観点の整理

- **担当者数が少なく、異動・氏名変更がまれ**  
  → ハードコーディングのメリット（オフライン・速度・シンプルさ）が活きやすい。
- **担当者が多く、台帳の更新が頻繁**  
  → API で都度台帳を参照する現状の方式の方が、整合性を保ちやすい。
- **折衷案**  
  - アプリ起動時や「設定」で一度だけ API から担当者一覧を取得し、ローカルにキャッシュする。  
  - キャッシュがあればオフラインでも表示し、オンライン時だけ更新する。

---

## 現在の利用箇所（参考）

- **担当者コード入力** … コード入力 → API で氏名取得 → 保存
- **伝票一覧・伝票詳細** … `handlerName` が無い伝票は `GET /api/employees` で氏名取得
- **伝票リンク画面** … 同様に担当者コード→氏名の解決で API 使用

ハードコーディングに切り替える場合は、上記の「API で担当者を取得している箇所」を、アプリ内の担当者マスタ（ハードコーディング or ローカルキャッシュ）参照に置き換える形になります。

---

## 50人分で重くなるか

**重くなりません。**

- 50件の「コード → 氏名」は **Map で数 KB 程度**。アプリ全体から見れば無視できるサイズです。
- 参照は **Map のキー検索（O(1）** なので、件数が 50 でも 500 でも体感差はありません。
- 起動時の読み込みも一瞬で終わります。

---

## おすすめ：ハイブリッド方式

**アプリに約50人分を組み込み、一覧にないコードだけ API で取得する**方式は、現状の API（コード指定の1件取得のみ）のまま使える良いバランスです。

| 項目 | 内容 |
|------|------|
| **通常時** | 組み込みの Map を参照するだけなのでオフライン可・即時表示。 |
| **新規・未登録コード** | ローカルに無いときだけ `GET /api/employees?code=...` を呼ぶ。 |
| **取得結果のキャッシュ** | API で取ったコード⇔氏名をローカルに保存しておけば、次回からは API 不要（退職者などは台帳側で削除されれば 404 になるが、アプリ側キャッシュは「次回起動でクリア」など運用で対応可能）。 |

**実装の流れ**

1. **組み込みマスタ** … `Map<String, String>`（コード→氏名）を定数 or `assets/employees.json` で用意。50人分をここに格納。
2. **解決ロジック** … `resolveEmployeeName(code)` のような関数を1か所に用意。  
   - まず組み込み Map を検索 → あればそのまま返す。  
   - なければ（必要なら）メモリ/永続キャッシュを検索 → あれば返す。  
   - どちらにもなければ `ApiClient.fetchEmployee(code)` を呼び、取得できたらキャッシュに保存してから返す。
3. **呼び出し側** … 担当者コード入力・伝票一覧・伝票詳細・伝票リンクなど、現在 `fetchEmployee` を呼んでいる箇所を、上記の `resolveEmployeeName`（または `Future<Employee?>` を返すラッパー）に置き換える。

これで「社員が増えたり減ったり」しても、  
- **増えた人** … 初回は API、その後はキャッシュで対応可能。  
- **減った人** … 台帳から削除されれば API は 404。組み込みリストはリリース時に更新すればよい。

## その他のスマートな方法

| 方法 | 概要 | 利点 | 注意点 |
|------|------|------|--------|
| **オンデマンドキャッシュのみ** | ハードコーディングは使わず、初回だけ API で取得し、結果をローカルにキャッシュ。2回目以降はキャッシュだけ見る。 | アプリに名簿を持たない。新規担当者も1回取得すれば以後オフライン可。 | 初回は必ずオンライン必要。キャッシュの期限や上限（例：100件で古いのから削除）を決める必要あり。 |
| **起動時に一覧取得** | API に「全件取得」を追加（例: `GET /api/employees` で50人一覧）。アプリ起動時に1回だけ呼び、結果をメモリ or 永続キャッシュに保存。 | 常に台帳と一致。ハードコーディング不要。 | API の改修が必要。オフライン起動時は前回キャッシュがあればそれを使う運用にする。 |
| **JSON アセット** | 50人分をソースコードではなく `assets/employees.json` で持つ。アプリは起動時にこれを読み込み。 | 名簿の更新時にコードを触らず JSON だけ差し替えてビルドできる。 | 更新のたびにアプリ再配布は同じ。 |
| **ハイブリッド（上記）** | 組み込み50人 + 無いときだけ API + 取得結果をキャッシュ。 | オフラインで大半が即時。新規は API で対応。API 変更不要。 | 組み込みリストの更新はリリースに含める必要あり（年1回などで十分なら問題になりにくい）。 |

---

## オンデマンドキャッシュについて

**オンデマンドキャッシュ**は、「担当者名が必要になったタイミングでだけ API を呼び、取得した結果をローカルに保存しておく」方式です。アプリに最初から名簿（ハードコーディング）は持たず、**使われたコードの分だけ**キャッシュが増えていきます。

### 動き方

1. **初回（キャッシュにないコード）**  
   担当者コードから氏名を求めるとき、まずローカルのキャッシュを検索する。  
   → 無い → `GET /api/employees?code=...` で API を呼ぶ。  
   → 取得成功なら「コード → 氏名」をキャッシュに保存してから返す。  
   → 失敗（オフライン・404 など）ならそのままエラー。

2. **2回目以降（キャッシュにあるコード）**  
   同じコードのときはキャッシュを参照するだけ。  
   → API は呼ばない。オフラインでも氏名を表示できる。

つまり「必要になったとき（オンデマンド）に API で取得し、その結果をキャッシュする」ので、**オンデマンドキャッシュ**と呼びます。

### キャッシュの置き場所

- **メモリ（Map）** … アプリ起動中だけ有効。終了すると消える。実装が簡単。
- **永続化（SharedPreferences / ファイル）** … アプリを再起動しても残る。オフラインでも前回取得した担当者を表示できる。
- **両方** … 起動時に永続キャッシュを読み込み、参照はメモリの Map で行い、API で新規取得したらメモリと永続の両方に書く。よくあるパターン。

### メリット

- **アプリに名簿を持たない** … 担当者台帳は API（DB）だけが正とする。アプリの再配布で名簿を更新する必要がない。
- **新規担当者も1回取得すればよい** … 台帳に追加された人は、誰かがそのコードを初めて使ったときに API で取れ、以後はキャッシュで表示できる。
- **使った分だけキャッシュが増える** … 実際に登場するコードだけ保存するので、50人全員を毎回取る必要はない（全件取得 API がなくても現行の「コード指定1件」API で足りる）。

### デメリット・注意点

- **初回はオンライン必須** … まだキャッシュにないコードは、ネットが通っているときでないと氏名を取得できない。オフライン初回は「コード: 123」のままなどになる。
- **キャッシュの設計が必要**  
  - **有効期限（TTL）** … 何日か経ったら無効にして再取得するか。氏名変更に追従させるなら短め、変動が少なければ長め or なし。  
  - **件数上限** … 放置するとキャッシュが増え続けるので、例として「最大 100 件で、古いものから削除」などにするとよい。  
  - **退職者** … 台帳から削除されると API は 404。キャッシュに残っている間は古い氏名が表示され続ける。TTL や「404 のときはキャッシュから削除」で緩和可能。

### まとめ（オンデマンドキャッシュ）

- 名簿をアプリに持たず、**必要になったときだけ API で取得し、その結果をローカルにキャッシュする**方式。
- 新規担当者も「誰かが初めてそのコードを使ったとき」に取得され、以後はオフラインでも表示できる。
- 初回はオンラインが必要で、キャッシュの有効期限や件数上限を決めておくと運用しやすい。

## 担当者キャッシュ（初回一括 + 永続 + メモリ + オンデマンド）検証手順

実装後、次のシナリオで手動確認することを推奨する。

| シナリオ | 手順 | 期待結果 |
|----------|------|----------|
| 初回起動・オンライン | API を起動し、アプリを初回起動。ホーム表示後しばらく待つ。 | 担当者一覧がバックグラウンドで取得され、永続キャッシュに保存される。SnackBar は出ない。 |
| 初回起動・オフライン | ネットを切り、キャッシュ未作成の状態でアプリを起動。 | 「担当者一覧の取得に失敗しました。オンライン時に再試行します。」の SnackBar が表示される。 |
| 2回目起動・オフライン | 1回目でキャッシュ取得済みの状態でネットを切り、アプリを再起動。担当者コード入力で 1〜101 のコードを入力。 | キャッシュから即時で氏名が表示され、保存できる（API は呼ばれない）。 |
| 範囲外コード・オンライン | 担当者コード入力で 1〜101 以外のコード（例: 150）を入力。 | オンデマンドで API が呼ばれ、台帳にいれば氏名が表示・保存される。次回からはキャッシュで表示される。 |
| 伝票一覧の担当者表示 | 伝票一覧を開き、担当者名が無い伝票がある場合。 | キャッシュにあれば即時表示。無ければオンデマンド取得後に表示が更新される。 |

---

## まとめ

- **50人分のデータ量は気にしなくてよい**です。
- **ハイブリッド（組み込み + 無いときだけ API + 取得結果キャッシュ）** は、現行 API のまま運用でき、オフラインと新規担当者の両方に強いおすすめ案です。
- API を拡張できるなら **「全件取得」+ 起動時キャッシュ** も、台帳一元管理という意味でスマートな選択肢です。
